# Prompt: Generate an epic.md from a plan document

Use the following prompt with any LLM to generate a well-formed epic.md file
from your plan.md document. The epic.md must conform to the format expected
by fry.sh.

## Prerequisites

Before running this prompt, you should have:
1. A `plans/plan.md` file with your holistic build plan, OR a `plans/executive.md`
   file with executive context (at least one is required; if only executive.md is
   provided, plan.md can be auto-generated by fry-prepare.sh)
2. The `epic-example.md` file (as a formatting reference)
3. Optionally, both files (plan.md for the build plan, executive.md for alignment context)
4. Optionally, a `.fry/AGENTS.md` file (auto-generated by fry-prepare.sh if missing)

## The Prompt

Copy everything between the --- markers below and paste it into your LLM,
attaching your plan.md and epic-example.md as context.

---

You are a senior software architect. Your job is to decompose a build plan
into a sequenced set of sprint phases for an autonomous AI coding agent.

The output is an `epic.md` file that will be consumed by a bash script called
`fry.sh`. The script runs an AI coding agent (OpenAI Codex or Claude Code) in
an iterative loop — one sprint at a time, fresh context each iteration. The
agent reads its instructions from a file called `.fry/prompt.md` (assembled from the
epic), reads `.fry/sprint-progress.txt` for memory from previous iterations in the current sprint, reads `.fry/epic-progress.txt` for summaries of prior sprints, and reads
`plans/plan.md` for strategic context.

## Input Documents

**plan.md** (attached): The holistic build plan. This is your primary source
for WHAT to build. It may be in any format — prose, bullet lists, numbered
sections, tables. Extract the implementation details from whatever structure
it uses.

**epic-example.md** (attached): The FORMAT REFERENCE showing the exact
syntax and structure the output must follow. Study it carefully — especially
the @directive syntax and the 7-part sprint prompt anatomy.

## Your Task

Analyze plan.md and produce an epic.md file that:

### 1. Decomposes the plan into 4-10 sequential sprints

Follow this layering order (adapt to the specific project):

| Sprint Position | Layer | Example Content |
|---|---|---|
| 1 (always) | Scaffolding | Project init, config, Docker, DB pool, Makefile |
| 2-3 | Data foundation | Schema, migrations, seeds, domain types, interfaces |
| 4-6 | Implementation | Business logic, data access, engines/services |
| 7-8 | Integration | API handlers, routes, service wiring |
| N (always last) | Verification | DI wiring, integration tests, E2E smoke tests |

### 2. Right-sizes each sprint

Each sprint should:
- Produce 5-20 files
- Be completable in 10-35 iterations
- Be independently verifiable (you can write a concrete checklist)
- Have clear dependencies on earlier sprints (never on later ones)

If a layer from the plan would produce 30+ files, split it into two sprints.
If it would produce 1-3 files, merge it into an adjacent sprint.

### 2a. Applies effort-level sizing

If an `@effort` level is specified, it constrains sprint count and density:

| Level    | Max Sprints | Max Iterations | Prompt Detail | Notes |
|----------|------------|----------------|---------------|-------|
| `low`    | 2          | 10-15          | Concise       | Combine layers, skip scaffolding sprint |
| `medium` | 4          | 15-25          | Moderate      | Merge related layers |
| `high`   | 10         | 15-35          | Full 7-part   | Current default behavior |
| `max`    | 10         | 30-50          | Extended      | Add analysis + quality gate sections |

If no effort level is specified, auto-detect based on plan complexity:
- 1-3 files -> low
- 4-15 files -> medium
- 15+ files -> high

### 3. Writes specific, actionable sprint prompts

Each sprint prompt MUST follow this 7-part structure:

**OPENER**: "Sprint N: [What] for [Project]."

**REFERENCES**: "Read .fry/AGENTS.md, then plans/plan.md [§sections if applicable]."
Point the agent at the specific parts of the plan relevant to this sprint.

**BUILD LIST**: Numbered list with EXACT specifications:
- Exact file paths (e.g., `internal/repository/account.go`)
- Function/method signatures (e.g., `Upsert(ctx, projectID, input) (*Account, bool, error)`)
- SQL DDL details (column names, types, constraints, indexes)
- API shapes (method, path, request/response bodies)
- Algorithm details (thresholds, formulas, weights)

DO NOT write vague instructions like "create the account repository."
DO write specific instructions like "internal/repository/account.go —
AccountRepository with Upsert(ctx, projectID, input) using INSERT ON CONFLICT
DO UPDATE, metadata merge via JSONB || operator."

**CONSTRAINTS**: "CRITICAL: [thing that WILL go wrong if ignored]."
These come from patterns you observe in the plan — technology choices,
naming conventions, security requirements, things the agent will get wrong.

**VERIFICATION**: Reference .fry/verification.md for the concrete, machine-executable
checks that fry.sh runs independently. The sprint prompt should describe what
success looks like in prose and point to .fry/verification.md as the source of truth:
- "Verification checks are defined in .fry/verification.md (sprint N)."
- Include a brief bulleted summary of key outcomes for the agent's guidance.
NOT vague items like "code is clean" or "everything works."

**STUCK HINT**: "If stuck after N iterations: [most likely cause + fix]."
Think about what typically goes wrong at this layer:
- Sprint 1: import paths, Docker networking
- Schema sprints: FK ordering, migration sequence
- Domain/types: circular imports between packages
- Repository: SQL syntax for the specific DB driver
- Wiring/DI: interface type mismatches

**PROMISE**: "Output ===PROMISE: SPRINTN_DONE=== when [exit criteria]."
The exit criteria should match the verification checklist.

### 4. Configures global directives

Based on the plan, determine:
- `@epic`: A descriptive name
- `@engine`: Which AI engine to use (`codex` or `claude`, default: `codex`)
- `@docker_from_sprint`: Which sprint first needs Docker (usually 2)
- `@docker_ready_cmd`: Health check for the primary DB service
- `@require_tool`: All CLI tools the project needs (language runtime, docker, etc.)
- `@pre_sprint`: Dependency management command (go mod tidy, npm install, pip install, etc.)
- `@model`: Override the AI agent model (optional, alias: `@codex_model`)
- `@engine_flags`: Extra flags for the agent CLI (optional, alias: `@codex_flags`)
- `@max_fail_percent`: Maximum percentage of checks that can fail and still pass (optional, default: 20; 0 = strict)
- `@compact_with_agent`: Use AI agent to summarize sprint progress (optional, default: mechanical extraction)
- `@review_between_sprints`: Enable mid-build sprint review between sprints (optional, default: disabled)
- `@review_engine`: AI engine for the reviewer LLM session (optional, default: same as `@engine`)
- `@review_model`: Model override for reviewer (optional, default: same as `@model`)
- `@max_deviation_scope`: Maximum sprints a single deviation can touch (optional, default: 3)

### 5. Handles cross-sprint concerns

- Sprint 1 ALWAYS includes: ".fry/AGENTS.md already exists. Do NOT modify it."
- Each sprint states "CRITICAL: This is a [language] project. No [other language]."
- Later sprints reference what earlier sprints created
- The final sprint includes an exhaustive acceptance criteria checklist
  derived from the plan's requirements/success criteria

## Output Format

Produce ONLY the epic.md file content. No preamble, no explanation, no markdown
code fences around the output. The output should start with `#` (the epic
title comment) and be directly saveable as a file.

## Quality Checklist (verify before outputting)

- [ ] Sprints are numbered sequentially starting at 1
- [ ] Every sprint has @name, @max_iterations, @promise, @prompt
- [ ] Every prompt follows the 7-part structure
- [ ] Build lists use exact filenames and signatures, not vague descriptions
- [ ] Verification sections reference .fry/verification.md for concrete checks
- [ ] Dependencies only flow forward (Sprint 4 never depends on Sprint 5)
- [ ] Sprint 1 is scaffolding, final sprint is integration/E2E
- [ ] Global directives match the project's technology stack and chosen engine
- [ ] The promise token in the prompt text matches the @promise directive
- [ ] No sprint exceeds 35 max_iterations
- [ ] Total sprints are between 4 and 10

## Usage Workflow

The full workflow from plan to running build:

```
1. Author your input document(s) — at least one of:
   ├── plans/plan.md (detailed build plan — what to build)
   └── plans/executive.md (executive context — why to build it)
   If only executive.md is provided, plan.md is auto-generated by fry-prepare.sh.
   If both are provided, plan.md is the build plan, executive.md is the north star.

3. Generate .fry/AGENTS.md + epic (choose one):

   a) Automatic — just run fry.sh with a filename that doesn't exist yet:
      └── ./fry.sh epic.md --dry-run
      └── ./fry.sh epic.md --engine claude --dry-run  (use Claude Code)
      └── fry-prepare.sh is called automatically:
          Step 1: generates .fry/AGENTS.md from plan.md
          Step 2: generates .fry/epic.md from plan.md + AGENTS.md
          Step 3: generates .fry/verification.md from plan.md + epic.md

   b) Standalone — run fry-prepare.sh directly:
      └── ./fry-prepare.sh epic.md
      └── ./fry-prepare.sh epic.md --engine claude    (use Claude Code)

   c) Manual — use this prompt with any LLM:
      └── Attach: plans/plan.md + epic-example.md
      └── Paste the prompt from the "The Prompt" section above
      └── Save output as .fry/epic.md (create .fry/AGENTS.md separately)

4. Review and adjust the generated epic
   └── Check sprint sizing, verify specificity, adjust iteration counts

5. Validate
   └── ./fry.sh epic.md --dry-run

6. Run
   └── ./fry.sh epic.md                        (default: codex)
   └── ./fry.sh epic.md --engine claude         (use Claude Code)
   └── FRY_ENGINE=claude ./fry.sh epic.md       (via env var)
```

Step 5 is important — always review the generated epic. The LLM will get the
structure right but may miss nuances from your plan, merge things that should
be separate, or be too vague on certain specifications. A 10-minute review pass
saves hours of wasted iterations.

## Tips for Better Generation

- **Minimum viable starting point is `plans/plan.md` OR `plans/executive.md`.** If you
  only provide executive.md, the LLM generates plan.md with full design authority.
  The more detail you provide (in either file), the better everything downstream will be.

- The more specific your plan.md is, the better the epic will be. If your plan
  says "add a user table" the epic will be vague. If your plan includes column
  names, types, and constraints, the epic will include them too.

- If your plan has natural phase boundaries (Phase 1: Foundation, Phase 2:
  Features, etc.), tell the LLM which phase to generate the epic for.

- For very large plans (50+ pages), generate the epic in two passes: first ask
  for just the sprint decomposition (names, dependencies, rough scope), review
  that, then ask for the full epic with prompt details.

- If the generated epic has more than 8 sprints, look for sprints that can be
  merged. More than 8 sprints usually means the decomposition is too granular.

- If any sprint prompt is longer than 100 lines, it's probably trying to do
  too much. Split it.
